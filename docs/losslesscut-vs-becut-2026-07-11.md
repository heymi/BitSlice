# LosslessCut 与 BeCut：固定时长分片对比

调研日期：2026-07-11。LosslessCut 结论仅引自其[官网](https://losslesscut.app/)，BeCut 结论来自当前源码。

## 结论

LosslessCut 是一把面向视频处理熟手的「无损剪辑瑞士军刀」；BeCut 应该定位为「把一个视频按指定时长，快速、清楚地切成可交付片段」的 Mac 原生工具。对固定时长切片这个单一任务，BeCut 的理想流程更短；对任意区间、不同容器和多轨素材，LosslessCut 明显更完整。

## 固定时长切片工作流

| 环节 | BeCut 当前实现 | LosslessCut |
| --- | --- | --- |
| 核心入口 | 单文件导入后自动按 `segmentDuration` 计算全部片段；用户只需调时长、名称、格式、分辨率、目录再导出。 | 同一工具承载修剪、删除、合并、轨道处理、remux、CSV/EDL 等场景；适合复杂任务，但首次用户需要理解切点、片段和导出模式。 |
| 预览 | 播放器、播放速度、静音、逐帧步进、当前自动分片的可视化和点击跳转。 | 官网列出可缩放时间线、帧/关键帧跳转、缩略图和音频波形。 |
| 分片规则 | 只支持一条等时长规则，从 0 开始连续切到结尾；最后一段可短于设定时长。 | 可创建并标记多个任意片段，导入/导出 CSV/EDL；不只是等时长。 |
| 导出 | MP4/MOV/M4V；所有精确分片均使用 AVAssetReader/Writer 重编码，保留源帧率、方向，并按兼容性回退编码。 | 直接复制数据流实现无损裁切，且可在兼容格式间 remux；可处理视频、音频、字幕等轨道。 |

## BeCut 在这个场景的优势

1. **单一目标更直接。** 当前界面把「分段时长」放在最前，随后是命名、格式、分辨率、导出位置；目标是把长视频快速拆成均匀片段，不要求用户先理解剪辑项目或各种导出模式。
2. **结果在导出前可见。** 自动显示片段数量、每段时长、最后一段长度和首个文件名，并可点选片段预览其起点。这比手动逐段打点更适合上传时长受限的平台、课程/录屏分发等重复工作。
3. **交付参数更少但更显性。** MP4/MOV/M4V、原始/720p/1080p/2K/4K、目标目录和文件名前缀是同一屏的少量决定项；输出目录不可写与空间不足会先报错。
4. **分辨率变换是一项真实能力。** BeCut 在需要时重编码并保留原帧率与画面方向；LosslessCut 的主张是快速无损流复制，两者不是同一个交付取向。

## BeCut 的不足与风险

1. **不能把自己说成无损。** LosslessCut 官方明确：无损切点会向前取最近关键帧，开始位置可能早于指定点；这正是流复制的边界。BeCut 当前选择精确重编码来避免关键帧回退，但会产生编码成本与画质取舍；首发也只对经过验证的容器/编码组合做兼容性说明。
2. **只支持自动等时长，不能手动编排。** 没有添加/拖拽/删除单个切点，不能混合长短片段，也没有按文件大小分割。这是 LosslessCut 的片段面板、标签和 CSV/EDL 的直接能力差距。
3. **素材和轨道能力窄。** 当前只允许单个输入文件，导出容器限定 MP4/MOV/M4V；没有批量源文件、合并、轨道选择/提取、字幕、音频波形或 remux。复杂素材会更倾向 LosslessCut。
4. **项目可复用性不足。** BeCut 只持久化通用导出设置和最近文件；尚未保存某个视频的切片方案，也不能导入/导出分片清单。

## 优先级建议

不要复制 LosslessCut 的全部功能。先把 BeCut 的差异写实：`导入一个视频 → 选择 15/30/60/180 秒或自定义 → 预览片段数量 → 一键导出`。

随后补三个最能提高可信度的能力：

1. 如果未来加入快速直通模式，应明确提示关键帧可能让切点向前偏移，并继续保留当前精确模式。
2. 为常用平台提供 15/30/60 秒预设和「分片计划」摘要；这会比通用时间线编辑器更快。
3. 加入手动增删切点和分片方案保存。这样保留极简主路径，同时覆盖从固定分片走向轻量剪辑的自然需求。
